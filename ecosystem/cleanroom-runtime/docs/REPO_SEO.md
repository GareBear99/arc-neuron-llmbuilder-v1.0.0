# Repository SEO and GitHub Positioning

This document defines the public-facing naming and discoverability posture for the repository. The goal is to improve search relevance without making claims the codebase cannot support.

## Release title format

Use this format for GitHub releases:

`vX.Y.Z — ARC Lucifer Cleanroom Runtime`

## README positioning rules

Keep the README centered on what the repository actually contains:
- deterministic runtime spine
- receipts, replay, rollback, and policy tracking
- memory tiers and archive lineage
- bounded self-improvement
- local-model integration surfaces
- repair intelligence via FixNet

Avoid turning the repo into vague hype around AGI. Search strength is better when the wording is precise and repeatable.

## GitHub setup checklist

- set the repository description exactly
- add the recommended topics
- upload `assets/social-preview.svg` as the social preview image
- pin the repo on the profile if this is a flagship system
- create the first release from the matching tag
- enable Issues and Discussions if public feedback is desired
- add a short release summary for each version with concrete changes

## Off-GitHub discoverability

For external posts, lead with the most concrete angle instead of the broadest one. Good examples:
- deterministic local AI runtime for persistent operator workflows
- local-first GGUF/llamafile operator shell with replay and rollback
- Python clean-room runtime for agent memory, repair lineage, and bounded self-improvement
